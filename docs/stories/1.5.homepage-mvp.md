### Homepage Component Architecture

[Source: architecture.md#Repository Structure]

**File Structure for Story 1.5:**

```
app/
├── (marketing)/
│   └── page.tsx                 # Homepage (THIS STORY)
│       └── uses Server Component (no 'use client' needed for main page)
│       └── imports section components below
components/
├── sections/
│   ├── Hero.tsx                 # Hero section with CTA
│   ├── TrustBar.tsx             # Client logos
│   ├── ServicesPreview.tsx      # 4 service cards
│   ├── WhyKBCOM.tsx             # 3 benefits
│   ├── Stats.tsx                # Trust indicators (years, projects, clients)
│   └── FinalCTA.tsx             # Bottom CTA section
tests/
└── unit/
    ├── app/
    │   └── page.test.tsx        # Homepage integration test
    └── components/
        └── sections/
            ├── Hero.test.tsx
            ├── ServicesPreview.test.tsx
            └── (other section tests)
```

**Component Philosophy:**

- **Server Components First**: Homepage and all sections should be Server Components by default (better performance, SEO)
- **Client Components Only When Needed**: Only use 'use client' if component needs state, effects, or event handlers
- **Separation of Concerns**: Each section is a reusable component for maintainability
- **Atomic Design**: Sections are "Organisms" level in atomic design

[Source: architecture.md#Rendering Strategy]

**Rendering Strategy for Homepage:**

- **Static Site Generation (SSG)**: Homepage should be pre-rendered at build time
- **No ISR needed yet**: Content is hardcoded for MVP (will use CMS in Epic 3-4 for portfolio/blog sections)
- **Metadata**: Use `generateMetadata()` function for SEO optimization
